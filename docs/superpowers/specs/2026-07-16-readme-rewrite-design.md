# Dreams README 重写设计

## 目标

将项目名称统一写为 **Dreams**，参考 AGFS-MEM README 的技术项目结构，重新说明项目当前真正解决的两个问题：

1. 从用户与 AI 的长期日常对话中持续提纯用户，形成不断进化的 `USER.md` 人物画像。
2. 从日常任务中提取 AI 的决策经验，形成决策卡，并通过周期整理逐渐塑造稳定、拟人的 AI 决策身份。

README 不使用来源声明，不罗列无助于理解产品的技术组合，也不把尚未实现的“可调用用户 Skill”写成现有能力。

## 核心表述

### 用户提纯

用户画像不是简单的偏好列表。`USER.md` 可以逐步积累有证据支持的身份背景、兴趣喜好、表达习惯、工作方式、长期目标、约束和对 AI 的期待。它的目标是形成一个越来越清晰的用户人物形象。

`user_id` 隔离放在安全与服务边界章节说明，不作为用户提纯本身的定义。

### AI 提纯

AI 决策卡保存可复用的决策经验，包括使用场景、判断信号、决策原则、实际结果、适用边界和置信度。周期整理会合并重复卡片、归档过时内容，并提炼稳定的决策规则。

README 将“拟人”解释为 AI 逐渐形成连续、稳定、可追溯的判断方式，而不是模仿某种说话语气或进行角色扮演。

### 做梦机制

做梦是贯穿两类产物的后台更新过程：接收已完成对话，分类其中值得长期保留的证据，分别更新用户人物画像和 AI 决策卡，再定期整理两类产物。更新只从下一次任务开始生效。

## README 结构

README 按以下顺序组织：

1. `Dreams` 项目简介与两项核心能力。
2. 架构图：对话输入、后台分类、两类提纯产物、周期做梦、下一任务上下文。
3. 核心产物：`USER.md`、AI 决策卡和最终决策规则。
4. 项目结构：只列仓库中实际存在的主要模块。
5. API：对话写入、下一任务上下文、做梦与回滚接口。
6. 用户提纯流程。
7. AI 拟人化提纯流程。
8. 周期做梦与下一任务生效机制。
9. 上游对话接入：主动推送或 NDJSON 增量拉取。
10. 多用户服务边界、容错、快照和回滚。
11. 核心模块、配置与开发运行方式。

## 事实边界

- 当前用户产物是持续进化的 `USER.md`，不是完整生成的可调用 Skill。
- 当前 AI 产物是 Markdown 决策卡和整理后的决策规则。
- 不声称项目使用 Redis、mem0、Embedding 或向量数据库。
- 不把内部工作流名称包装为第三方技术。
- README 全文不出现用户禁止的项目名称及版本描述。
