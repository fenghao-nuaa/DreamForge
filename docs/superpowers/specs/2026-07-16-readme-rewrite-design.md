# Dreams README 重写设计

## 目标

将项目名称统一写为 **Dreams**，参考 AGFS-MEM README 的技术项目结构，重新说明项目当前真正解决的两个问题：

1. 从用户与 AI 的长期日常对话中持续提纯用户，形成不断进化的 `USER.md` 人物画像。
2. 从日常任务中提取 AI 的决策经验，形成决策卡，并通过周期整理逐渐塑造稳定、拟人的 AI 决策身份。

README 不使用来源声明，不罗列无助于理解产品的技术组合，也不把尚未实现的“可调用用户 Skill”写成现有能力。

## 核心表述

### 用户提纯

用户画像不是简单的偏好列表。`USER.md` 可以逐步积累有证据支持的身份背景、兴趣喜好、表达习惯、工作方式、长期目标、约束和对 AI 的期待。它的目标是形成一个越来越清晰的用户人物形象。

`user_id` 隔离放在安全与服务边界章节说明，不作为用户提纯本身的定义。

### AI 提纯

AI 决策卡保存可复用的决策经验，包括使用场景、判断信号、决策原则、实际结果、适用边界和置信度。周期整理会合并重复卡片、归档过时内容，并提炼稳定的决策规则。

README 将“拟人”解释为 AI 逐渐形成连续、稳定、可追溯的判断方式，而不是模仿某种说话语气或进行角色扮演。

### 做梦机制

AI 在每次交互中得到的信息通常是局部、零散和增量的。随着对话不断积累，用户信息会出现重复、变化和矛盾，AI 的决策经验也会散落在不同任务中，难以直接复用。

做梦是贯穿两类产物的后台综合过程：读取已有产物和过去的对话记录，合并重复信息，根据较新的证据修正过时结论，保留无法确认的冲突，并从多次交互中发现新的稳定模式。每次更新前保存回滚快照，整理结果只从下一次任务开始生效。

当前实现会更新磁盘上的产物，同时保留修改前快照和运行报告。README 不将其描述为“输入永不修改并生成完全独立的存储”。

### 用户人物画像做梦

输入是当前用户的 `USER.md`、该用户过去的对话记录和每条结论对应的来源证据。整理过程包括：

- 合并重复或同义人物信息；
- 根据较新的明确表达更新过时信息；
- 对无法确认的矛盾同时保留并标记；
- 提炼跨多次对话才显现的稳定人物特征；
- 保留身份、喜好、表达、目标、工作方式和互动模式。

输出是一个持续进化、带证据来源的 `USER.md` 新版本。

### AI 决策身份做梦

输入是已有 AI 决策卡、当前决策规则，以及过去任务中的判断过程和结果。整理过程包括：

- 提取场景、信号、原则、结果、适用边界和置信度；
- 合并表达同一原则的决策卡；
- 归档被新经验取代的重叠卡片；
- 从多张卡片中形成稳定的决策规则；
- 避免把一次偶然选择误写成 AI 的永久身份。

输出是可检查的 AI 决策卡、定期提炼的 `DECISION_RULES.md`，以及逐渐稳定的 AI 决策身份。

## README 结构

README 按以下顺序组织：

1. `Dreams` 项目简介与两项核心能力。
2. 架构图：已有产物与对话输入、后台分类、两类提纯产物、周期做梦、下一任务上下文。
3. 核心产物：`USER.md`、AI 决策卡和最终决策规则。
4. 项目结构：只列仓库中实际存在的主要模块。
5. API：对话写入、下一任务上下文、做梦与回滚接口。
6. 用户提纯流程。
7. AI 拟人化提纯流程。
8. 周期做梦与下一任务生效机制。
9. 上游对话接入：主动推送或 NDJSON 增量拉取。
10. 多用户服务边界、容错、快照和回滚。
11. 核心模块、配置与开发运行方式。

## 事实边界

- 当前用户产物是持续进化的 `USER.md`，不是完整生成的可调用 Skill。
- 当前 AI 产物是 Markdown 决策卡和整理后的决策规则。
- 当前做梦通过修改前快照和回滚保证可恢复，不声称生成完全独立的输出存储。
- 不声称项目使用 Redis、mem0、Embedding 或向量数据库。
- 不把内部工作流名称包装为第三方技术。
- README 全文不出现用户禁止的项目名称及版本描述。
